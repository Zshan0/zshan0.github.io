Operating systems
Any computer system has hardware, system software and application software.
In system software, the most important part is the operating system.
Operating system acts as a shell for other software on the hardware and acts as an abstraction layer.
Basic processor model
The instruction that gets executed depends on the program counter of the processor.
The basic fetch decode execute model is modified with an additional input called interrupt, whenever there is some signal from the hardware, there is an "interrupt" which takes place and changes the instruction that has to be performed.
To perform there is another routine called interrupt service routine which changes the flow of control in the system.
What happens when we switch on the system?
When the system is turned on, RAM contains no instructions to execute. so to start the processor in a sequence a small part of the RAM is made of ROM or flash-RAM.
not drawn to scale
Flash-RAM: It is non volatile part of the RAM which can be erased and re programmed, this contains the initial procedures to be executed while boot happens. Flash RAM has a micro loadar
When we switch on the computer, the PC starts fetching instructions from the flash ram and executes them. this is the first level boot-loader which will be present in the BIOS routines written in flash-ram along with the BIOS.
Main purpose of the 1st level boot-loader is to fetch the 2nd level boot-loader into the RAM present in the hard-disk which is more complex(example grub).
Memory map of a single tasking operating system
One of the popular examples of a single tasking operating system is MS-DOS
Process: Any task which is in execution.
Kernel: Integral part of the operating system which has the control over the whole system.
Issues with single tasking operating system:
It is single tasking OS. you can not do more than 1 process at a time.
If there is a bug which leads to computer hanging and also can end up overwriting the OS code if it consumes too much memory. The only way to fix this is a hard reboot.
Multi programmed OS in '65 - '80
This OS has the capacity to switch from one process to another on command. In the diagram we see Job B being used while Job A is writing to storage simultaneously.
This is normally how a program would run, initially when a program started executing it is said to be in user mode. In which system is performing instructions given by the user. when there is a read operation to be performed from the I/O devices, it switches to kernel mode. Where it follows the required instructions from the kernel to fetch from I/O devices.
But then the time taken for fetching the data from the I/O devices will be wasted, so the OS calls the schedule functions.
What the schedule function does is sees whats next to be executed on the CPU and schedules them for the execution(in the diagram it is program 2). when instructions of program 2 are executing, the OS goes to user mode. when the I/O devices are done reading, another interrupt is issued, interrupt service is again invoked and goes back to program 1 execution by suspending program 2 execution.
Limitation: This kind of system only works when there is no direct interaction with the user, that is a job is assigned to the computer, it completes it and returns the response rather than current systems where there is a lot of interactivity with real time response. why? because user interactivity leads to a lot of interrupts making the system inefficient due to rescheduling.
Time-sharing OS
All the jobs are allocated some slice of time to execute. after each time slice is complicated, there is a timer interrupt which results in rescheduling of the processes.
This gives a feeling of all the processes being active at the time.
But how does the processor decide what to schedule next? this is decided by Scheduling policy.
All the resources are shared among the processes.
Issue: since multiple jobs are present in the main memory at the same time, one can end up modifying the other jobs due to a bug.
The Illusion: We need to create an illusion for a job/process that the whole Processor and main memory is available for its execution. in other words we want to give the process its own virtual PC.
