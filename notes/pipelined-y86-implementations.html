Pipelined Y86 implementations
Basic idea is to take the processor which executes Y86 sequentially i.e SEQ and make changes to it by adding pipelined systems in it to make it more efficient.
Rearranging computation stages:
The order at which things are computed in SEQ are convenient but not efficient since results in final part of the cycle which can be computed beforehand might be needed for the next cycle to start.
One such thing is PC. we need the new PC value beforehand to start the next cycle for the next instruction. so if we could some how compute PC in the beginning of the clock cycle, that will be convenient.
We can see that PC is generated with the help of previous instructions, that is. it is not stored somewhere but generated dynamically. hence it can be generated early on.
p refers to previous.
Rearrangement of logic like this is called circuit re-timing.
Inserting pipeline registers
We insert registers in between the stages where we store the values of the variables calculated in previous stage.
The sequence is almost identical to the SEQ model, just that registers have been inserted.
What do the registers store?
F: Predicted value of the PC.
D: It is present between fetch and decode hence it stores the most recent value of fetched data.
E: It is present between decode and execute stages. holds the most recent decoded instruction ready for execution.
M: Holds the most recent executed instructions, it also stores information about branch conditions and other conditional statements.
W: Between memory stage and feedback, this supplies the results to registers and return address to the PC selection for the next cycle.
Relabeling the signals
If we observe we notice that there are multiple places where the signal is stored, even though they are the same signal, based on what step they are, they differ from what steps have been worked on them. for example stat. there are 4 places where stat is stored and each of them are diff. so mixing up the signals and using 1 for the purpose of the other would be a disaster.
Simple method to fix this is to add a prefix to the name based on what pipeline they are being stored, that is stat becomes D_stat, E_stat, M_stat, and W_stat.
But there are also signals which are generated inside a stage by a control block, for these lower case prefix is used, such as f_stat.
One more change from SEQ to PIPE- would be that in SEQ, the addresses generated for valE and valM, that is dstE and dstM which are generated in decode stage are directly given to the address inputs of address inputs of register file, but to avoid the mismatch of data and address from 2 diff instructions, they are carried along with the rest of the data through execute and memory stage.
Select A control block:
valA for the execute stage is either taken from the valP generated in decode stage or from A port of the register file.
Why is this the case?
The only instruction which requires valP in the memory stage is call, and the instruction which needs it in execute is the jump, which does not need to access any register reading, so we can reduce the amounts of things that are carried.
PC prediction:
The main goal was to start the cycle of a new instruction after every clock tick. but for this we need the next value of PC which in SEQ was generated at the end. In PIPE- we are computing it in the start but how?
We can predict what branch it will take as long as it is not a branch instruction in the fetch cycle itself.
We do always taken branch prediction here, but for ret instruction, we stop other instructions until ret value has been obtained.
