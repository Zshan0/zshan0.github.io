Architecture and protocols
Layers of architecture:
Perception layer:
Also known as device layer, this is the main layer, that is it contains physical objects and sensor devices.
This layer collects all the information by the sensor devices and send it network layer.
This is further transferred to middleware layer where it is processed securely.
Network layer:
Also called Transmission layer
Transmission medium can be both wire or wireless.
The layer supports requirements of latency, bandwidth and security.
Middle-ware layer:
Mainly responsible for service management.
The whole idea is that devices only connect to other devices which do similar things and provide similar services.
This is where the database lies, this is where the data processing happens and decisions are made.
Application layer:
The applications based on the information provided by the middleware layer get global access here.
smart home, smart farming etc are implementations lying in this level.
Business layer:
management of overall IoT system including the applications.
This is where the monitoring happens to determine what actions to be taken.
IoT protocols:
Transmission control protocol is one of the old one protocols responsible for data transfer which rely upon byte streams both the ways.
Interchange of messages:
Pub/Sub approach:
The client which is sending information about something, he is referred to as a publisher. and the one receiving information is subscriber. But the one intervening between them is a broker.
MQTT: Runs over TCP/IP
its a lightweight pub/sub approach which can even run on mobiles.
This is used in remote areas where only small footprints of code and hardware is allowed.
Example being satellites, home automation.
This is preferred for mobiles because of low battery consumption, small size and minimized data packets.
The structure of topics in MQTT are like files in filesystems, they are separated by a comma.
wildcards: + - the sublevel - - the whole subtree.
QoS:
This is protocol which specifies the delivery requirements, so Retransmission is a common thing to do to guarantee the delivery.
QoS 0 - that means there is no guarantee of the message reaching the recipient since it is only transmitted once. this is called fire and forget.
QoS 1 - guarantees that the message will be delivered at least once, it can be sent multiple times, the receiver sends a DUP signal indicating that it has already received it.
What is PUBack? it is sort of an acknowledgement for the signal received.
QoS 2 - exactly once, this guarantees that the signal is received only once.
PUBREC is sent back once when the signal is received, the client can discard data. and then it sends back PUBREL stating that it received the PUBREC so that the broker can discard all the data about PUBREC and so on till PUBCOMP.
Data Distributive Service:
This is a protocol for data centric connectivity.
Low latency, extreme reliability data.
This is commonly used in smart cities, grids, robotics, autonomous vehicles.
DDS Abstraction:
There is a virtual data space provided for each operation, this is called global data space abstraction.
The application only stores what data needs and where it needs
Reading and writing happens asynchronously and independently.
DDS isolates and guides each writer to its data reader from the rest of the network topology and connectivity. This means that the reader need not worry about the rest of the server or where the reader is since DDS will anyway guide the data to it.
DDS also has the capacity to do it while run time, allowing plug and play.
It knows what offers and wants what.
Since 1 part is isolated, a single point failure does not lead to crash of the whole system.
Dynamical adaptation allows it to choose the best path to share data.
Data streams are in terms of topics
Topics: <name, type, QoS>
Information Scopes:
All the DDS information lives within a domain which is organized into partitions.
These can be considered as broker, subscriber and publisher domains.
DDS is able to store last n samples of the stream data, this is based on its durability.
Volatile: No durability
Transient: Data availiable till the supplier is running
Durable: Available till the system is running.
DDS delivers a sub-sequence of things written against topic since middle elements can get dropped because of network loss when it is doing best effort.
Last n values condition: There is a guarantee that last n values will definitely be delivered written for a topic. This is bad for slow consumers.
MQTT v/s DDS
MQTT is centralized where as DDS is de-centralized architecture, MQTT has 1 common broker where as in DDS it is isolated paths.
MQTT is more focused on delivery of the message rather than security, reliability and fault tolerance as in the case of DDS.
MQTT is less complex compared to DDS.
