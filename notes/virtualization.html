Virtualization
The goal of doing virtualization is to provide an illusion to processes being performed on the computer of them having the whole processor and the whole main memory for themselves to use.
Physical addressing
A basic system such as IoT devices use physical addressing. The whole memory is considered to be 1 large array of bytes and is mapped by numbering them(physical address), the CPU directly uses the mapped number to access the data required.
Virtual addressing
The CPU generates a Virtual address which is then sent to Memory management unit(MMU) which then maps it to the physical address which are then supplied back to the CPU.
Address spaces
Linear address space: ordered set of contiguous non negative integers
{0, 1, 2, 3...}
Virtual address space: Set of virtual addresses (up to to 2^n)
Physical address space: Set of physical addresses (up to to 2^m)
When a new process is added, it gets its virtual space where it is meant to operate, the main idea behind this is to implement isolation between processes.
Why Virtual memory?
1.simplifies the memory management. how? Well all the processes get the same virtual space which can later be mapped to diff parts of the physical address, there is no chance of mapping two programs to the same place. (This is mind blowing). All the programs can work from 0 to 2^n without overlapping each other.
2. Isolating the processes becomes simple, MMU has to map different virtual space to different physical space.
3. DRAM acts as a cache (more on that further down).
Virtual memory for caching
Virtual memory and the physical memory, both are divided into logical pieces called pages which are usually in the order of kilobytes. virtual pages can be put under 3 categories, cached, uncached, unallocated.
What happens when the CPU wants some data from and it gives some address to MMU?
We can draw analogy between Virtual memory and the physical memory to the concept of cache and main memory.
Suppose CPU wants some data, MMU locates which virtual page does that memory fall into, after knowing the virtual page, it checks if that virtual page has been allocated in the physical memory. If it has been allocated, this is considered as a page hit else it is considered as a page miss.
How to deal with page miss? we assign a physical page to the virtual page, this can be considered as promoting the virtual page to physical memory.
what if all the physical pages have been occupied by some or the other virtual page, we have to evict some virtual page from the physical memory based on page eviction policy.
Translation between virtual address to physical address
When the CPU generates an address for some data it wants to read/write, the address generated is a virtual address.
That virtual address is supplied to MMU, in MMU for every process there is a respective page table. This page table maps a virtual page to a physical page.
This page table is present in the main memory, MMU uses this page table to figure out if the the virtual page is present in the DRAM (physical memory) or in the disk.
Where in disk?
The virtual page can be present in 2 areas on the disk.
Swap area: Its basically a substitute for the physical memory when the physical memory does not have enough space. This helps the computer pretend to have more RAM than it has.
Placeholder: Wherever the program is present on the disk itself can be the mapped area, that is suppose we have a program which we want to run, but there isn't enough space on physical memory bring all of its components. so only a piece is brought to main memory.
Page fault: When CPU tries to access data present in a page which has not been mapped to a page in physical memory, this results in a page fault. this is overcome by bringing the relevant data to physical memory by evicting some page or by terminating the process
In the above case, accessing page 3 or 6 results in a page fault.
While mapping some page from disk to physical memory, fully associative mapping is followed

Virtual memory for 2 processes
Suppose 2 processes are currently running, and they need to access some data and instructions, it will be stored in the physical memory.
We see that 2 processes can map to 1 single physical page even with different virtual pages.
But the one mapped to the same physical page should be read only. so that could contain instructions from the library both of the processes need.
How does the mapping happen?
With the help of page table we can see which virtual page is mapped to which physical page.
Every memory access gets mapped to the page table and the actual physical address. this makes the process slow.
But this is overcome by using a translation table called Translation Look aside Buffer (TLB).
So what this buffer does is it caches the mapping from virtual to physical page address in the processor, so if 1 page is being accessed again and again it stores it in the cache and hence speeds up the conversion.
Translating the address itself based on the page table
The offset is same for both Physical and Virtual address since the size of the pages is same, it will split it in such a way that the starting p bits of the address can remain unchanged.
The page numbers can be converted from the virtual number to the physical number with the help of a page table.
Valid bit: the bit tells whether the virtual page has been allocated or not.

Page Fault handling
Suppose the instruction is accessing data from a virtual page that is not mapped, this is detected based on the valid bit present for the page.
1.
The message is sent to the OS indicating the missing page number(trap)
2.
OS locates the missing page from the swap area or the executable
3.
the page is then brought to the physical memory in an empty physical page
4.
then the page table is informed about the new mapping which changes/resets the table
5.
the instruction is again restarted
Note: Number of page faults can be measured with the help of the PERF tool
The problem:
Suppose the main memory is of 4GB and each page is of 4KB, there will 2^20 or else 1MB of pages, and if each entry is of a byte size, 8 bytes will be required for the page table of each process. This might not be a lot but if the main memory size gets bigger then this leads to a problem.
The solution:
We use 2 mapping, first we use 1 mapping to figure out in which section is the address for the required page is present. and then from there we do final mapping.
One major advantage this idea has that the whole page table is not necessary, we can use and provide the inner page table only of the required size preventing the wastage of space.
Why does Virtual memory work?
The main cause which helps in virtual memory being so efficient is the principle of locality.
Suppose there are n processes running on the system
Case 1: working size of all the processes combined < main memory
All the data required for the processes can be stored on the main memory
Case 2: working size of all the processes combined > main memory
This case is called thrashing there is a drop in performance due to continuous page swapping both ways.
